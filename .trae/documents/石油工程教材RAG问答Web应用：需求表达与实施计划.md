## 你该怎么“提要求”
- **先定目标与边界**：一句话目标（给谁用/解决什么问题/成功标准），再写清楚MVP必须做什么、明确不做什么。
- **用用户故事表达**：例如“作为学生/老师，我想上传教材并提问，以便快速定位教材出处与页码”。
- **把需求拆成可验收的条目**：每条都要能测试（输入→输出→验收标准），避免“体验要好/尽量快”这类不可验证描述。
- **把风险前置**：PDF解析质量、公式/表格、OCR、版权合规、长文档耗时、费用控制、并发与限流。

## 需不需要写 PRD
- **建议写一个轻量PRD（1–3页）**：尤其你要同时涵盖前端、后端、RAG、LLM多供应商、上传解析与索引、权限与费用控制；PRD能把范围冻结，避免实现时反复返工。
- **不必“产品级完整PRD”**：如果你是自研/小团队，采用“PRD Lite + API契约 + 数据模型草图”就够了。

## PRD Lite 模板（直接照填）
1. **背景/目标**：
   - 目标：面向石油工程教材的问答，回答必须可追溯引用（页码/章节/段落）。
   - 成功标准：例如 Top-k 引用命中率、回答可用性、平均响应时间、单问成本。
2. **用户与场景**：学生/教师/管理员；典型问题类型（概念解释、步骤、对比、公式/表格）。
3. **范围**：
   - MVP：上传电子教材→解析→建索引→对话问答→展示引用→管理文档。
   - Out-of-scope：OCR（第一期）、多租户（第一期）、复杂权限（第一期）。
4. **核心流程**：上传→解析切分→向量化→检索→生成→引用展示；失败与重试策略。
5. **功能需求（可验收）**：
   - 文档：支持PDF（MVP），最大大小、页数；上传后能看到“处理中/完成/失败”。
   - 切分：按章节/段落（策略可配置）；保留页码/标题等元数据。
   - 检索：向量+关键词混合；可配置topK。
   - LLM：支持 OpenAI/Anthropic/DeepSeek/Qwen/Ollama 之一先打通，再扩展到多供应商；支持选择模型与温度。
   - 结果：回答+引用列表（文档名/页码/片段）+“在原文中打开”。
6. **非功能**：
   - 性能：上传解析异步；问答P95耗时目标。
   - 成本：token预算与缓存策略；并发限流。
   - 安全：上传文件存储、鉴权、日志脱敏。
7. **数据与合规**：版权声明/仅自用；数据保留与删除。
8. **验收清单**：用10条真实教材问题做验收集（含标准引用）。

## 技术实施计划（按你指定技术栈）
### 1) 基础架构与工程化
- 前端：Vue 3 + TypeScript + Vite + Naive UI；页面路由、状态管理、API封装、错误提示。
- 后端：FastAPI；Pydantic模型；统一异常与日志；OpenAPI文档。
- 数据库：PostgreSQL + pgvector（用于向量检索）；迁移工具（如Alembic）。

### 2) 文档上传与索引流水线（RAG Ingestion）
- 存储：本地文件系统（按doc_id目录）+ DB记录；不使用Docker。
- 解析：PDF文本抽取（优先纯文本；表格/图片先不做OCR）；生成page-level元数据。
- 切分与嵌入：可配置chunk大小/重叠；选择嵌入模型（先选一个供应商跑通）。
- 入库：chunks写入pgvector表；支持重建索引与删除。
- 异步任务：FastAPI后台任务或独立worker（后续可升级）；提供进度查询。

### 3) 问答链路（Retrieval + Generation）
- 检索：向量topK + 可选BM25/关键词过滤；返回带页码的候选片段。
- 生成：LangChain/LangGraph组织流程（检索→重排序可选→回答→引用格式化）。
- 可信输出：强制引用；无足够证据时返回“无法在教材中定位”。

### 4) 多LLM供应商适配层
- 设计统一Provider接口：chat、embeddings（可选）、模型列表、超时/重试。
- 先实现1个云端（如OpenAI或Qwen）+ 1个本地（Ollama）作为对照，再逐个扩展。

### 5) Web UI
- 页面：登录/（可选）→ 文档管理（上传、状态、删除、重建索引）→ 问答页（对话、引用、跳转原文）→ 设置页（选择LLM与参数）。

### 6) 验证与交付
- 单元测试：解析/切分/检索/引用格式。
- 端到端：上传一份PDF样例→索引→10条验收问题。
- 性能与成本：记录token与耗时指标，形成最小监控面板或日志统计。
